# Walking Debug Flow

This project now uses `debug.py` as the single live walking debug entrypoint.
Do not edit files inside `.venv/Lib/site-packages`; those belong to Python packages.

## Run

```powershell
python debug.py
```

`debug.py` sends one algorithmic walking phase at a time:

1. `01 shift-to-<support>-support`
   - Moves ZMP/CoM toward the support foot.
   - Support roll is generated by `compute_pose` from `com_y`, `support_leg`,
     `ankle_roll_gain`, and `hip_abduct_gain`.
   - No per-phase PWM injection is used.

2. `02 lift-<swing>-leg`
   - Swing foot lift is generated by `_lift_profile`.
   - Forward travel is generated by the foot trajectory relative to the stance foot.
   - Thigh, knee, and ankle pitch are generated by IK from the swing-foot
     target. There is no extra thigh-boost PWM layer.

3. `03 land-<swing>-leg`
   - The swing foot lands at the planned target X.
   - The foot reaches its forward target before the vertical drop finishes.
     Landing therefore lowers the foot at the new step position instead of
     retracting thigh/knee toward the original pose.
   - The landing target is constrained by `landing_gap_mm`, so touchdown keeps
     a minimum X distance from the support foot.
   - Landing roll release is generated in `walking_engine.py`, not manually patched.

## Tuning Policy

Prefer algorithm-level parameters in `debug.py::Config`:

- `zmp_support_ratio`: how far the weight shifts toward the support foot.
- `hip_abduct_gain`: hip counter-roll for keeping the torso organized.
- `ankle_roll_gain`: support-foot roll response to lateral CoM shift.
- `step_height`: swing-foot clearance.
- `step_x_ratio`: forward travel scale.
- `left_swing_x_scale` / `right_swing_x_scale`: side-specific step reach.
- `landing_gap_mm`: minimum forward/backward touchdown distance from the
  support foot.
- `left_step_height_scale` / `right_step_height_scale`: side-specific lift height.
  Current walking debug keeps left/right equal so cycle 2 mirrors cycle 1.
- `swing_advance_end_phase`: when the swing foot must already reach target X.
  Keep this earlier than `lift_end_phase` so landing is a vertical drop at
  the new foot placement.
- `arm_swing_pwm`: continuous opposite arm swing. During walking, arms do not
  share the same physical direction. The walking engine generates one logical
  envelope: left-leg swing is positive, right-leg swing is negative. Hardware
  direction config maps it to servos `8` and `17`. Current hardware mapping
  keeps both shoulder swing channels with positive direction in walking.
  During the landing/lowering segment, both arm offsets blend back to `0`.

Current phase-1 support-shift defaults are intentionally visible:

- standing thigh bias: servo `4 = 1444`, servo `21 = 1556`,
- `zmp_support_ratio = 0.88`
- `hip_abduct_gain = 0.35`
- `ankle_roll_gain = -0.38`

This means phase 1 should clearly move the support ankle/hip roll pair:

- right support: servo `1` and `5`,
- left support: servo `24` and `20`.

Approximate current phase-1 target:

- support ankle roll: about `3.8 deg`, around `43 PWM`,
- support hip counter-roll: about `3.5 deg`, around `39 PWM`.
- right support: servo `1` about `+43 PWM`, servo `5` about `+39 PWM`,
- left support: servo `24` about `+55 PWM`, servo `20` about `-75 PWM`.
- phase-2 support ankle returns near neutral: right servo `1` about `1490`,
  left servo `24` about `1510`; support hip keeps the phase-1 compensation.
- sagittal pitch directions are mirrored in PWM: right-leg pitch servos
  `2/3/4` and left-leg pitch servos `23/22/21` now use the same positive
  IK direction, so right swing should match left swing instead of folding
  backward.
- During right-leg touchdown, `compute_pose()` mirrors the right pitch group
  `2/3/4` around `STANDING` so `land-right-leg` matches the proven
  `land-left-leg` sagittal direction while keeping roll `1/5` unchanged.
- phase-3 landing gap X: at least `16 mm` from the support foot.

The swing ankle roll stays neutral so phase 2 does not look like it changes
lean direction.

Avoid per-phase manual PWM edits such as:

- changing servo `4` or `21` only inside phase 1,
- forcing one support phase to look correct by adding a hardcoded PWM offset,
- making separate debug-only pose logic that does not exist in `walking_engine.py`.

If a phase looks wrong, fix the generator in `src/walking_engine.py`, then let
`debug.py` capture the resulting pose from the engine.
